## 1. Orphan ref layout

`refs/aria/coordination` is a **git orphan ref** — its commit tree is completely isolated from
the repository's code history. This ref stores only coordination metadata; it never touches
source or docs.

### 1.1 Directory structure inside the ref

```
refs/aria/coordination (orphan tree)
├── claims/
│   └── <container-id>/
│       └── <session-id>.yaml        # active / yielded claim (one file per writer)
└── archive/
    └── <YYYY-MM>/
        └── <container-id>/
            └── <session-id>-<claimed_at>.yaml   # GC-moved done claim (7-day retention)
```

### 1.2 Key invariants

| Invariant | Rationale |
|-----------|-----------|
| **File-per-writer partitioning**: each session writes only `claims/<own-container-id>/<own-session-id>.yaml` | Zero-conflict push — two containers can never write the same file path simultaneously |
| **Append-only within a session's file**: a session only ever overwrites its own claim path, never another's | `git push` on this ref is always a fast-forward add or in-place replace of exactly one file |
| **archive/ is write-once**: once a done claim is GC'd to `archive/`, it is never rewritten | Audit trail integrity |

### 1.3 Bootstrap (幂等)

On first use, `TASK-012` (`coordination_ref.py::bootstrap`) creates the orphan ref via
`git commit --orphan` with an empty tree. Subsequent bootstrap calls detect the ref exists
and are no-ops. If the remote ref does not yet exist, the first `push` creates it.

## 2. Claim YAML schema (v1)

### 2.1 Field table

| Field | Type | Required | Semantics | Valid values / example |
|-------|------|----------|-----------|------------------------|
| `schema_version` | string | YES | Schema version for forward-compatibility; reader must handle unknown version by downgrading | `"1"` (current) |
| `track_id` | string | YES | Deterministic work ID, 1:1 bound to a branch / OpenSpec change; normalized via `TASK-014` `derive()` function | `"multi-terminal-coordination"` |
| `owner` | string | YES | git `user.email` local-part (everything before `@`); used in collision classification (cross-owner vs self-multi-container) | `"creationhikari"` |
| `container` | string | YES | `~/.aria/container-id` persistent short-UUID + optional human label, fallback to hostname (generated by `TASK-011`) | `"devbox-A"` / `"a3f7c2d1"` |
| `session` | string | YES | Ephemeral session identifier (short random + start timestamp); unique within a container run | `"s-7f3a@0931"` |
| `phase` | string | YES | Ten-step-cycle phase at time of claim write or last heartbeat | `"A"` / `"B.2"` / `"D.3"` — same enum as Layer H frontmatter |
| `status` | string | YES | Claim lifecycle state | `"active"` / `"yielded"` / `"done"` / `"unknown"` (see §3) |
| `claimed_at` | string | YES | UTC ISO 8601 timestamp (seconds precision) when the claim was first written; **immutable after initial write** — reconcile uses this for "earliest wins" | `"2026-05-20T09:31:00Z"` |
| `heartbeat_at` | string | YES | UTC ISO 8601 timestamp (seconds precision) of the most recent heartbeat push; updated every `HEARTBEAT_INTERVAL` (600 s) and on phase transition | `"2026-05-20T11:01:00Z"` |
| `superseded_from` | string | NO | Populated during reconcile take-over: `"<owner>/<container>/<session>"` of the claim being superseded | `"creationhikari/devbox-B/s-2b9e@1104"` |

### 2.2 Status enum

| Value | Meaning | Transitions |
|-------|---------|-------------|
| `active` | Session is actively working on this track; heartbeats expected | → `yielded` (collision reconcile) / → `done` (Phase D release) |
| `yielded` | Session has voluntarily or by reconcile stepped back; track is free to re-claim | → `active` (same session re-acquires) / → `done` |
| `done` | Work is complete; claim may be GC'd to `archive/` after 7 days | terminal (GC moves to archive, tombstone sha256 retained) |
| `unknown` | Reader encountered a claim file with an unrecognized `schema_version`; content is preserved but the claim is skipped by reconcile (safety-first) | read-only sentinel assigned by reader, never written by a live session |

### 2.3 Minimal complete example

```yaml
schema_version: "1"
track_id: multi-terminal-coordination
owner: creationhikari
container: devbox-A
session: s-7f3a@0931
phase: B.2
status: active
claimed_at: "2026-05-20T09:31:00Z"
heartbeat_at: "2026-05-20T11:01:00Z"
```

### 2.4 Example with optional field (reconcile take-over)

```yaml
schema_version: "1"
track_id: multi-terminal-coordination
owner: simonfishgit
container: laptop
session: s-9c21@1310
phase: B.2
status: active
claimed_at: "2026-05-20T09:31:45Z"
heartbeat_at: "2026-05-20T11:05:00Z"
superseded_from: "creationhikari/devbox-B/s-2b9e@1104"
```

### 2.5 Archived claim example (GC)

File path: `archive/2026-05/devbox-A/s-7f3a@0931-2026-05-20T09:31:00Z.yaml`

```yaml
schema_version: "1"
track_id: multi-terminal-coordination
owner: creationhikari
container: devbox-A
session: s-7f3a@0931
phase: D.3
status: done
claimed_at: "2026-05-20T09:31:00Z"
heartbeat_at: "2026-05-20T17:55:00Z"
```

A sha256 tombstone digest of this file's content is retained after GC for audit purposes.

## 3. schema_version evolution

### 3.1 Current version lock

`schema_version: "1"` is locked. The field set defined in §2.1 is the v1 contract.
No new required fields may be added to v1 (doing so would break existing readers).

### 3.2 Reader downgrade on unknown version

When a reader encounters a claim file with `schema_version` other than `"1"`:

1. The reader **must not crash** or raise an unhandled exception.
2. The reader **must return a `ClaimRecord` with `status="unknown"`** (a sentinel value).
3. `status="unknown"` claims are **skipped by reconcile** (see §4.2 — safety-first:
   a claim whose semantics are opaque must not influence verdict).
4. The `unknown` status is never written by a live session; it is exclusively a reader-side
   sentinel. It does not appear in `STATUS_ENUM` for write validation.
5. The reader **should emit a `soft_error("claim_schema_unknown_version", ...)`** so operators
   can detect schema drift across containers running different plugin versions.

### 3.3 Introducing v2

Any field additions, semantic changes, or removals require:
- A new `schema_version` value (`"2"`)
- A new OpenSpec Spec proposal (at least Level 2)
- Updated reader logic in `claim_schema.py` with explicit version dispatch

## 4. Edge cases

### 4.1 Matrix

| Scenario | Detection | Reader behavior | Reconcile behavior |
|----------|-----------|-----------------|--------------------|
| `schema_version` missing | `KeyError` on `raw["schema_version"]` | `parse_claim` returns `None` + `soft_error("claim_schema_invalid")` | Skipped (no ClaimRecord produced) |
| `schema_version` unknown (e.g. `"2"`) | `raw["schema_version"] != SCHEMA_VERSION_CURRENT` | Returns `ClaimRecord(..., status="unknown")` | Skipped — `unknown` claims do not participate in verdict |
| Required field missing (any of the 8) | `KeyError` in field extraction | `parse_claim` returns `None` + `soft_error("claim_schema_invalid")` | Skipped |
| Field type wrong (e.g. `schema_version: 1` integer) | Type check fails | `parse_claim` returns `None` + `soft_error("claim_schema_invalid")` | Skipped |
| `status` not in `STATUS_ENUM` (write-side values only) | Enum check | `parse_claim` returns `None` + `soft_error("claim_schema_invalid")` | Skipped |
| Corrupt YAML (unparseable bytes) | `yaml.YAMLError` in caller (`TASK-013`) | Caller catches, skips file, emits `soft_error("claim_yaml_parse_error")` | Skipped — no `parse_claim` called |
| `claimed_at` / `heartbeat_at` not valid ISO 8601 | `datetime.fromisoformat` raises | `parse_claim` returns `None` + `soft_error("claim_schema_invalid")` | Skipped |
| `superseded_from` malformed (present but wrong format) | Format check `owner/container/session` | `parse_claim` returns `None` + `soft_error("claim_schema_invalid")` | Skipped |

### 4.2 `status="unknown"` in reconcile (safety-first)

Reconcile (`TASK-015`) **skips** any claim with `status="unknown"` entirely:

- It does not count as an `active` claim (no "active count" increment).
- It does not trigger collision detection (no verdict emitted for this claim).
- It does not affect verdicts for other claims on the same `track_id`.
- The claim file is preserved on disk — it is not deleted or archived prematurely.

Rationale: a claim whose schema version is unknown may encode semantics incompatible
with the current reconcile rules (e.g. v2 may have changed `claimed_at` semantics).
Silently treating it as `active` could cause incorrect yield decisions. Skipping is the
conservative, safety-first choice.

### 4.3 Clock skew

When two claims on the same `track_id` have `claimed_at` timestamps that differ by less than
`CLOCK_SKEW_WARN_THRESHOLD` (30 s, defined in `lib/constants.py` by `TASK-018`), reconcile
**downgrades to `CONFLICT` status** and defers to human decision rather than silently applying
"earliest wins". See `TASK-017` for board rendering of this case.
