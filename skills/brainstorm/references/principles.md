## 1. 一次只问一个问题 (One Question at a Time)

### 为什么重要

- 避免信息过载
- 让用户专注思考
- 便于追踪对话历史
- 防止 AI 急于下结论

### 实施要点

```yaml
正确:
  AI: 这个功能的主要目的是什么？
      a) 提高用户参与度
      b) 减少操作步骤
      c) 其他 (请说明)

错误:
  AI: 这个功能的目的是什么？目标用户是谁？有什么约束条件？
      (一次问三个问题，用户难以回答)
```

### 特殊情况

如果某个话题需要深入探索，拆分为多个连续问题：

```yaml
示例:
  Q1: 这个功能的主要目的是什么？
  A1: 减少操作步骤

  Q2: 具体想减少哪些操作的步骤？
  A2: 登录和配置流程

  Q3: 现在这些流程有多少个步骤？
  A3: 登录 3 步，配置 5 步
```

## 2. 优先多选题 (Multiple Choice Preferred)

### 为什么优先

- 用户选择更容易
- 减少歧义
- 便于统计分析
- 加快对话节奏

### 多选题设计原则

```yaml
选项数量: 3-5 个
覆盖范围:
  - 包含常见场景
  - 包含"其他"选项
  - 每个选项互斥

示例:
  AI: 用户角色有哪些？
      a) 管理员
      b) 普通用户
      c) 访客
      d) 其他 (请说明)
```

### 开放式问题的使用时机

```yaml
使用开放式问题当:
  - 选项无法预知
  - 需要详细描述
  - 需要用户原创想法

示例:
  AI: 请描述一下理想的使用体验？
```

## 3. 探索替代方案 (Explore Alternatives)

### 方案对比框架

| 维度 | 问题描述 | 评分标准 |
|------|---------|---------|
| 实现难度 | 开发工作量 | 简单/中等/复杂 |
| 用户体验 | 用户满意度 | 高/中/低 |
| 维护成本 | 后期维护难度 | 低/中/高 |
| 约束匹配 | 是否满足约束 | 是/否 |
| 风险评估 | 潜在风险 | 高/中/低 |

### 展示方案格式

```markdown
## 方案 A: {名称}

**描述**: {1-2 句话描述}

**优点**:
- {优点1}
- {优点2}

**缺点**:
- {缺点1}
- {缺点2}

**约束匹配**: {满足/违反} {约束名}
```

### 推荐逻辑

```yaml
推荐步骤:
  1. 分析每个方案的优缺点
  2. 考虑约束条件的限制
  3. 权衡短期和长期影响
  4. 给出推荐并说明理由

推荐格式:
  "我推荐方案 {X}，因为..."
  - 理由 1
  - 理由 2
  - 风险说明
```

## 4. 分段验证 (Incremental Validation)

### 分段原则

```yaml
每段长度: 200-300 词
分段依据:
  - 架构概述
  - 核心组件
  - 数据流
  - 错误处理
  - 测试策略

验证方式:
  - "这样对吗？"
  - "需要调整吗？"
  - "有疑问吗？"
```

### 分段验证流程

```
┌─────────────────────────────────────────────────────┐
│  展示段 1 (架构概述, 200-300 词)                     │
│                    ↓                                │
│  验证: "这样的架构安排看起来对吗？"                    │
│                    ↓                                │
│  ┌─────────────┬─────────────┐                      │
│  │ 用户确认    │ 用户有疑问   │                      │
│  │     ↓       │     ↓       │                      │
│  │  继续下一段  │  回退澄清    │                      │
│  └─────────────┴─────────────┘                      │
│                    ↓                                │
│  展示段 2 (核心组件, 200-300 词)                     │
│                    ↓                                │
│  验证...                                            │
└─────────────────────────────────────────────────────┘
```

### 常见问题处理

```yaml
用户说"不懂":
  → 回退到上一段
  → 换个方式解释
  → 用更简单的语言

用户说"不对":
  → 了解哪里不对
  → 调整方案
  → 重新展示该段

用户说"太快":
  → 减缓节奏
  → 增加验证点
  → 分更多小段
```

## 5. 灵活回退 (Be Flexible)

### 回退场景

```yaml
任何时候可以回退:
  - 用户对之前的假设提出质疑
  - 发现理解有偏差
  - 约束条件发生变化
  - 发现更好的方案

回退方式:
  - "让我回退一下..."
  - "我们重新考虑..."
  - "之前的假设可能不对..."
```

### 回退示例

```
用户: 等等，刚才说数据不出境，但其实可以用云服务

AI: 好的，让我回退一下。
    如果数据可以使用云服务，那我们的技术选型会有变化...
    [重新展示方案]
```
